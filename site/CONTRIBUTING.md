# Contributing to Everything Agents

Most additions to this site are content, not code. The site reads its
tutorials from a folder structure on disk, so adding lessons usually means
dropping in some MDX files and editing one JSON file. This guide walks you
through every common change.

## Adding a new topic

A *topic* is a sequenced set of lessons around one real agent. Each topic
lives at `content/topics/<topic-slug>/`.

### Step 1 — create the folder

```
content/topics/<topic-slug>/meta.json
```

`meta.json` looks like:

```json
{
  "slug": "<topic-slug>",
  "title": "Full topic title used in the page H1",
  "shortTitle": "Short title used in breadcrumbs",
  "oneLine": "One sentence describing the topic.",
  "prereqs": "Optional one-liner about who this is for.",
  "status": "available"
}
```

`status` values:

- `available` — listed on the home page in the active section, lessons render.
- `upcoming` — listed under "In the queue", but no lessons render even if MDX
  files exist.
- `draft` — hidden from listings.

### Step 2 — add it to the static registry

Open `lib/registry.ts` and add an entry to the `topics` array matching the
`meta.json` you just wrote. The static registry is what the home page and
topic index render at the moment; the MDX content loader (`lib/content.ts`)
also reads `meta.json` for the lesson route. Keep both in sync.

### Step 3 — add lessons

See "Adding a lesson" below. With at least one lesson and `status: "available"`,
the topic renders at `/topics/<topic-slug>` with all its lessons live.

## Adding a lesson

Lessons are MDX files at `content/topics/<topic-slug>/<NN>-<lesson-slug>.mdx`.
The leading number determines order. The slug after the number is the URL
fragment.

### Frontmatter

Every lesson starts with frontmatter:

```mdx
---
number: 4
title: Anatomy of a turn
keyIdea: One-sentence summary that appears under the H1 and as the SEO description.
minutes: 5
---
```

- `number` — display order, also the lesson number shown to readers.
- `title` — the lesson's H1.
- `keyIdea` — a one-sentence summary. Shows on the topic landing under the
  lesson title and is used for `<meta name="description">`.
- `minutes` — rough reading-time estimate.

### Body

Plain Markdown / GFM works. The MDX pipeline has these custom components
available globally:

- `<KeyIdea title="...">` — accent-rule callout for the lesson's central
  idea.
- `<Callout type="pitfall|note|warning" title="...">` — variant callouts.
- `<KnowledgeCheck question="...">` with `<Option correct>...</Option>`,
  more `<Option>` children, and one `<Explanation>` — end-of-lesson quiz.
- `<Mermaid caption="...">` — wrap a Mermaid source string in backticks.
  Theme-adaptive client render.
- `<Term definition="...">term</Term>` — inline glossary tooltip.
- `<Steps>` and `<Step title="...">` — numbered, anchored step list.
- Bespoke illustrations: `<SpectrumIllustration />`, `<TwoFailureModes />`,
  `<ControlSurfaceStack />`, `<TurnPipeline />`, `<DevLoop />`,
  `<TraceTimeline />`, `<RefundAgentCutaway />`, `<HallucinationLeak />`,
  `<DefenseOnion />`, `<LineageConvergence />`.

GFM tables, fenced code (Shiki-highlighted), and headings work normally.
Heading anchors are auto-generated by `rehype-slug`.

### Knowledge checks

Aim for one per lesson where a conceptual check fits naturally. Don't force
them onto every lesson; the glossary and the appendix don't need quizzes.
Always include an `<Explanation>` so readers learn from wrong answers.

```mdx
<KnowledgeCheck question="Which surface enforces a hard precondition?">
  <Option correct>
    `available when` — the gate is structural, not a hint to the LLM.
  </Option>
  <Option>
    `instructions:` — the prompt asks the LLM to be careful.
  </Option>
  <Explanation>
    Prompts are guidance. Gates are guarantees.
  </Explanation>
</KnowledgeCheck>
```

## Adding an illustration

Illustrations live at `components/illustrations/<name>.tsx`. They are
hand-authored React components returning inline SVG, theme-adaptive via
`currentColor` and CSS variables.

### Style rules (match existing illustrations)

- Use `currentColor` for default strokes.
- Use `oklch(var(--accent))` only for emphasis — the warm amber is the one
  voice for "this is the important part."
- Prefer a `viewBox` height around 160–320; readers shouldn't have to scroll
  to see the whole diagram.
- `<title>` and `<desc>` accessibility tags at the top of the SVG.
- Inter for labels, JetBrains Mono only for code-shaped strings.
- No animation, no decoration, no mascots.

### Wiring

After writing the component, register it in `components/mdx/components.tsx`:

```ts
import { MyIllustration } from "@/components/illustrations/my-illustration";
// ...
export const mdxComponents = {
  // ...existing...
  MyIllustration,
};
```

Then drop `<MyIllustration className="my-6 w-full" />` into a lesson MDX.

## Design system

Tokens live in `app/globals.css` as CSS variables. Tailwind reads them via
`tailwind.config.ts`. The full design language lives in `DESIGN.md` —
typography, color tokens, spacing, motion, accessibility targets,
microcopy defaults, and what we deliberately don't have.

If you're tempted to add a new accent color, marketing-style hero, or
animated illustration, read `DESIGN.md` first.

## Search, sitemap, SEO

These are all generated from your content automatically.

- `app/api/search-index/route.ts` calls `lib/search-index.ts` at build time
  to produce a JSON index of every available lesson. Adding a lesson
  automatically adds it.
- `app/sitemap.ts` reads `getAllTopics()` and emits `/sitemap.xml` at build
  time, including every static page, topic landing, and lesson.
- Every lesson's `generateMetadata` exports a description (the `keyIdea`
  frontmatter), an `openGraph` block, a Twitter card, and a canonical URL.
  You don't need to write SEO code per lesson — just write a good
  `keyIdea`.

## Running locally

```
pnpm install
cp .env.example .env.local   # fill in Clerk keys
pnpm dev
```

Open http://localhost:3000.

## Building for production

```
pnpm build
pnpm start
```

The build generates static pages for every available lesson, plus
`/sitemap.xml`, `/robots.txt`, the OG image, and the search index API.

## Deploying

The site deploys to Vercel as a Next.js project. `vercel.json` pins the
framework and adds a `www → apex` redirect. Required environment variables:

- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`
- `CLERK_SECRET_KEY`

Set them in Vercel's dashboard. The site renders without them; the sign-up
modal just won't open.

## Code conventions

- TypeScript strict mode is on.
- React Server Components by default; client islands marked with
  `"use client"` at the top.
- Components are small and one-purpose. Don't add props that aren't used.
- Tailwind classes are inlined; no per-component CSS modules unless
  there's a clear reason.
- New routes always export `generateMetadata` for SEO.
